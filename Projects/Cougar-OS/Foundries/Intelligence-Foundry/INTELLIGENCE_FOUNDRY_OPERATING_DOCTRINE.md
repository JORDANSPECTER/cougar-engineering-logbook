# Purpose

The Operating Doctrine defines how the Intelligence Foundry functions on a daily operational basis.

It establishes repeatable operational procedures for collection, verification, governance, monitoring, correction, and executive intelligence delivery.

While the Constitution defines permanent rules and the Blueprint defines architecture, this document defines operational behavior.

---

# Operating Mission

The Intelligence Foundry continuously transforms external information into trusted organizational intelligence while preserving governance, transparency, and explainability.

Daily success is measured by intelligence quality rather than collection volume.

---

# Daily Operational Cycle

The Foundry continuously repeats the following lifecycle:

```text
Review Collection Schedule
        ↓
Create Collection Missions
        ↓
Execute Collection Agents
        ↓
Verify Intelligence
        ↓
Create Intelligence DNA
        ↓
Evaluate Organizational Impact
        ↓
Update Enterprise Graph
        ↓
Update Memory Foundry
        ↓
Notify Executive Lanes
        ↓
Archive Mission History
```

---

# Source Operations

Every collection source has an operational state.

Allowed states:

- Proposed
- Under Review
- Approved
- Active
- Suspended
- Retired

Only Active sources may participate in scheduled collection.

---

# Collection Scheduling

Collection schedules define:

- execution frequency
- assigned Collection Agent
- owning Executive Lane
- retry policy
- timeout policy
- execution priority
- operational windows

Schedules may execute:

- hourly
- daily
- weekly
- event-driven
- manually

---

# Mission Operations

Every collection execution becomes a Collection Mission.

Each mission records:

- mission identifier
- assigned agent
- assigned Lane
- source
- timestamps
- retries
- completion status
- execution duration
- failures
- generated Intelligence DNA

Mission history is permanent.

---

# Verification Operations

Verification must occur before organizational promotion.

Verification evaluates:

- source reliability
- corroborating evidence
- contradictory evidence
- confidence
- freshness
- completeness

Verification outcomes remain permanently attached to Intelligence DNA.

---

# Intelligence DNA Operations

DNA records are created immediately after successful verification.

DNA operations include:

- creation
- confidence updates
- relationship creation
- correction history
- supersession
- retirement

DNA becomes the permanent record of external organizational knowledge.

---

# Organizational Impact Review

Every Intelligence DNA record is evaluated for organizational significance.

Possible outcomes:

- Evidence Only
- Memory Update
- Graph Node Update
- New Graph Node
- Mission Trigger
- Executive Alert
- Deferred
- Ignored

Graph growth is intentionally controlled.

---

# Enterprise Graph Operations

The Enterprise Organization Graph receives only approved mutations.

The runtime is responsible for:

- node creation
- relationship creation
- node updates
- synchronization
- graph refresh
- runtime notifications

Direct graph modification is prohibited.

---

# Memory Operations

Memory Foundry receives:

- supporting evidence
- historical observations
- contextual information
- executive intelligence
- organizational lessons

Memory preserves understanding rather than raw information.

---

# Failure Operations

Operational failures are classified.

Categories include:

- Source Failure
- Authentication Failure
- Network Failure
- Parsing Failure
- Verification Failure
- Runtime Failure
- Authorization Failure

Every failure records:

- cause
- timestamps
- retry count
- affected mission
- affected source
- recommended action

# Retry Policy

Recoverable failures may be retried.

Retry policies should define:

- maximum retries
- retry interval
- escalation threshold
- suspension criteria

Repeated failures may suspend a source until reviewed.
