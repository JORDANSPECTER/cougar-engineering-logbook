# Documentation Standards

---

# Purpose

Documentation is a permanent engineering asset.

It preserves engineering knowledge, explains architectural decisions, and enables future engineers to understand the platform.

Documentation is created alongside engineering work—not after it.

---

# Documentation Philosophy

Engineering work is not complete until it has been documented appropriately.

Documentation should explain:

- What was built
- Why it was built
- How it fits into the platform
- Future considerations

---

# Documentation Levels

## Daily Engineering Log

Purpose

Capture daily engineering progress.

Contents

- Mission
- Completed Work
- Challenges
- Lessons Learned
- Next Mission

---

## Release Documentation

Every release includes:

- README
- CHANGELOG
- Release Notes
- Milestones

---

## Architecture Documentation

Architecture documents explain:

- System purpose
- Relationships
- Design decisions
- Long-term direction

---

## ADRs

Architecture Decision Records capture significant engineering decisions.

Each ADR should document:

- Context
- Decision
- Alternatives
- Consequences

---

## Handbook

The Engineering Handbook defines permanent engineering standards.

These documents evolve slowly and guide future development.

---

# Documentation Principles

Documentation should be:

- Clear
- Accurate
- Current
- Maintainable

Avoid documenting implementation details that change frequently.

Focus on preserving engineering knowledge.

---

# Engineering Rule

If future engineers cannot understand why a decision was made, the documentation is incomplete.
